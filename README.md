

# HAYA-TOKU（早得クーポン）PHP PoC

## 概要

HAYA-TOKU は、**時間経過に応じて割引価値が変動する「早得クーポン」** を扱う PHP + PostgreSQL ベースの PoC です。

このPoCの目的は、単なる固定割引クーポンではなく、

- 早いタイミングほどお得に見える
- 時間経過に応じて割引率が変動する
- その変動を表示し、利用行動を前倒しさせる
- 利用時点の割引率を確定し、記録・分析できる

という **「時間 × 割引 × 行動誘導」** の構造を実装・検証することです。

---

## このPoCで採用する基本仕様

### 1. 未使用クーポンの割引率は流動する

クーポンは取得・発行した時点で価値が固定されるのではなく、**未使用の間も時間経過に応じて割引率が変動**します。

### 2. 割引率は使用時点で確定する

実際に店舗で利用した瞬間の割引率を、そのクーポンの最終的な利用割引率として確定します。

- 未使用中は流動
- 使用時点で確定
- 確定値を `used_discount_rate` として保存

### 3. 日次ベースで割引率を扱う

PoC段階では、割引率の変動は **日次ベース** を基本とします。

理由：

- 仕様説明がしやすい
- 管理画面で扱いやすい
- UI表示に落とし込みやすい
- 早得の意図（「あと何日で下がる」）を自然に表現できる

### 4. 早得の本質は「変動を見せること」

このPoCでは、単に割引率を変えるだけでなく、

- 現在割引率
- 残日数
- 次回変化タイミング
- 今使う理由になる文言

を表示できる構造を重視します。

# 責務分離（最重要）

本プロジェクトは以下の分離を正本とします。

```text
db.php              → 取る
coupon_logic.php    → 判断する
coupon_schedule.php → 計算する
dashboard / index   → 見せる
api                 → つなぐ
---

## 1. 取得・保存

### `lib/db.php`

担当：

- クーポンプラン取得
    
- 単一プラン取得
    
- 現在プラン取得
    
- クーポン取得
    
- DB保存
    
- DB更新
    
- DB正規化
    

ルール：

- 業務判断は入れない
    
- 画面表示用の文言は入れない
    
- 取得・保存に専念する
    

---

## 2. 業務判断

### `lib/coupon_logic.php`

`coupon_logic.php` は、**数値計算そのものではなく、業務上の状態判断**を扱います。

担当：

- 状態判定（draft / scheduled / active / ended / invalid）
    
- 表示可能判定
    
- 表示対象として選択可能か判定
    
- 利用可否判定
    
- 現在フロントに表示すべきプラン解決
    
- 利用不可理由コード
    
- 利用不可メッセージ
    
- 画面/API向け view model 構築
    

代表関数：

- `plan_status_code()`
    
- `plan_status_label()`
    
- `plan_can_be_selected_for_front()`
    
- `plan_can_be_displayed_now()`
    
- `resolve_current_display_plan()`
    
- `coupon_is_usable()`
    
- `coupon_use_denied_reason()`
    
- `coupon_use_denied_message()`
    
- `build_plan_view_model()`
    

ルール：

- 判断の正本はここ
    
- dashboard や API に同種の判定を重複定義しない
    
- CSS class や表示専用装飾は持たない
    

---

## 3. 数値計算

### `lib/coupon_schedule.php`

`coupon_schedule.php` は、**保存済みプランに対して、いま現在の割引率を計算する実行ロジック**です。

担当：

- 開始日時と終了日時から総日数を出す
    
- 現在日時から経過日数を出す
    
- 現在割引率を計算する
    
- 残日数を出す
    
- 次回いつ割引率が下がるかを出す
    
- 使用時点の割引率を確定する
    
- schedule snapshot を返す
    

ルール：

- 数値計算だけを担当する
    
- 業務状態コードの判定は持たない
    
- DB保存責務は持たない
    

---

## 4. 描画

### `admin/dashboard.php`

担当：

- プラン一覧表示
    
- 状態表示
    
- 表示対象のラジオ表示
    
- 編集導線
    
- UI上の文言表示
    
- UI用 class の切り替え
    

ルール：

- 状態判定の本体は持たない
    
- `coupon_logic.php` の結果を使って表示する
    
- 選択可否の判断本体は持たない
    
- 画面都合の class 変換だけを持つ
    

### `public/index.html`

担当：

- 現在クーポン表示
    
- 発行UI
    
- 使用UI
    
- モーダル制御
    
- APIレスポンス表示
    

ルール：

- クーポン状態の正本は持たない
    
- API が返した結果を描画する
    
- 業務判断をフロント側に重複実装しない
    

---

## 5. 入口・橋渡し

### `api/*.php`

担当：

- リクエスト受付
    
- 必要な取得・判断・計算の呼び出し
    
- レスポンス返却
    

方針：

- 業務判断を直接書かない
    
- `coupon_logic.php` / `coupon_schedule.php` / `db.php` を呼ぶ薄いAPIにする
    

---

## 状態コード（正本）

状態コードの正本は `coupon_logic.php` とし、以下へ統一します。

|code|意味|
|---|---|
|draft|下書き|
|scheduled|公開前|
|active|公開中|
|ended|終了|
|invalid|設定不正|

旧名称は使用しません。

- `inactive` → `draft`
    
- `upcoming` → `scheduled`
    
- `expired` → `ended`
    

---

## 表示構造（重要）

表示は以下の3要素で分離します。

```text
is_active → 有効候補（人の意思）
時間条件 → 状態（自動判定）
ラジオ   → 表示対象（選択）
```

この3つは混ぜません。

---

## 表示対象ルール

- フロント表示対象として選択可能なのは **active のみ**
    
- 判定の正本は `coupon_logic.php` の `plan_can_be_selected_for_front()` とする
    

---

## 表示対象決定ロジック

現在フロントに表示するプランは `resolve_current_display_plan()` で決定します。

優先順位：

1. dashboard で選択されたID
    
2. ただし active の場合のみ採用
    
3. 不適合なら `find_current_plan()` にフォールバック
    

---

## view model

表示用データの整形は `build_plan_view_model()` にまとめます。

これにより、

- dashboard
    
- API
    
- 今後の別画面
    

で、同じ状態情報を使い回せるようにします。

## モデルとロジックの分離

### `lib/coupon_model.php`

`coupon_model.php` は、**クーポン作成時の理想値（推奨初期値）を算出するモデル**です。

担当：

- クーポン作成時の推奨値を出す
    
- 初期割引率 / 最低割引率の候補を考える
    
- 粗利最大化に近い推移を試算する
    
- 利用率や利益面の観点から設計支援を行う
    

やらないこと：

- 今日の割引率を直接決める
    
- 残日数を計算する
    
- 次回いつ下がるかを計算する
    
- 使用時点の割引率を確定する
    

つまり、**実運用中の現在割引率を決めるファイルではなく、設計時の推奨値・理想値を考えるためのモデル**です。

---

## データ構造の基本方針

### 1. `coupon_plans`

クーポンの設計ルールを持つテーブルです。

例：

- タイトル
    
- 商品名
    
- 説明
    
- 開始日時
    
- 終了日時
    
- 初期割引率
    
- 最低割引率
    
- 利用条件
    
- メモ
    
- 有効/無効
    

ここに保存されるのは、**クーポンの実体ではなく、早得ルールそのもの**です。

### 2. `coupons`

発行されたクーポン実体を持つテーブルです。

例：

- クーポンコード
    
- 対応するプランID
    
- 発行日時
    
- 使用日時
    
- 使用時割引率
    

重要点：

- **使用時点で確定した割引率を保存する**
    

### 3. `usage_logs`

利用処理の記録を持つテーブルです。

例：

- 利用日時
    
- 利用クーポンID
    
- 利用時割引率
    
- 店舗ID
    
- スタッフID
    
- ユーザー識別子
    
- 補足メモ
    

このログは、後の分析や検証の基礎になります。

### 4. `view_logs`（将来または段階導入）

閲覧記録用のテーブルです。

PoC段階では必須ではありませんが、将来的には

- 閲覧数
    
- 利用率
    
- 割引率別利用傾向
    

を見るために追加候補です。

## APIの役割

### `api/coupon_current.php`

現在表示すべきクーポンプランを取得し、表示用情報を返すAPI。

### `api/coupon_issue.php`

クーポンを発行するAPI。

- クーポンコード生成
    
- クーポン実体作成
    
- 発行日時保存
    

### `api/coupon_get.php`

発行済みクーポンの状態取得API。

- クーポンコード/IDに紐づく状態取得
    
- 現在の利用可否確認
    
- 必要に応じた表示情報返却
    

### `api/coupon_use.php`

クーポン使用を確定するAPI。

最重要処理：

- 使用可能か確認する
    
- `coupon_schedule.php` で使用時点の割引率を算出する
    
- その値を確定する
    
- `used_discount_rate` に保存する
    
- 使用済み状態に更新する
    
- 利用ログを書き込む

## 管理画面の考え方

管理画面は、**固定割引クーポンを作る画面ではなく、早得ルールを設定する画面**として扱います。

主な目的：

- プラン作成
    
- 推奨値モデルを参考にした初期値入力
    
- 公開期間設定
    
- 初期割引率 / 最低割引率設定
    
- 商品名 / 利用条件 / 説明の設定
    
- 有効/無効切替
    
- フロント表示対象の選択
    

管理画面で扱うのは、**使用済み値そのものではなく、時間変動ルール**です。

## 今後の整理方針

コード構成は、最終的に以下のような分担を目指します。

- `coupon_model.php`  
    = 推奨値モデル（理想値・推奨初期値の算出）
    
- `coupon_schedule.php`  
    = 現在割引率・残日数・次回変化の計算
    
- `coupon_logic.php`  
    = 発行可否・利用可否・状態判定・表示対象解決
    
- `db.php`  
    = DB接続 + `coupon_plans` / `coupons` 基盤関数群
    
- `dashboard.php` / `public/index.html`  
    = 描画と操作
    
- `api/*.php`  
    = 入口と橋渡し
    

この分担により、**意味がぶれず、説明しやすく、段階的に整理しやすい構造**にしていきます。
