# HAYA-TOKU（早得クーポン）PHP PoC

## 概要

HAYA-TOKU は、**時間経過に応じて割引価値が変動する「早得クーポン」** を扱う PHP + PostgreSQL ベースの PoC です。

このPoCの目的は、単なる固定割引クーポンではなく、

- 早いタイミングほどお得に見える
- 時間経過に応じて割引率が変動する
- その変動を表示し、利用行動を前倒しさせる
- 利用時点の割引率を確定し、記録・分析できる

という **「時間 × 割引 × 行動誘導」** の構造を実装・検証することです。

---

## このPoCで採用する基本仕様

### 1. 未使用クーポンの割引率は流動する

クーポンは取得・発行した時点で価値が固定されるのではなく、**未使用の間も時間経過に応じて割引率が変動**します。

### 2. 割引率は使用時点で確定する

実際に店舗で利用した瞬間の割引率を、そのクーポンの最終的な利用割引率として確定します。

- 未使用中は流動
- 使用時点で確定
- 確定値を `used_discount_rate` として保存

### 3. 日次ベースで割引率を扱う

PoC段階では、割引率の変動は **日次ベース** を基本とします。

理由：

- 仕様説明がしやすい
- 管理画面で扱いやすい
- UI表示に落とし込みやすい
- 早得の意図（「あと何日で下がる」）を自然に表現できる

### 4. 早得の本質は「変動を見せること」

このPoCでは、単に割引率を変えるだけでなく、

- 現在割引率
- 残日数
- 次回変化タイミング
- 今使う理由になる文言

を表示できる構造を重視します。

## システムの責務分離

このPoCでは、責務を以下のように分けます。

### 推奨値モデル（`lib/coupon_model.php`）

`coupon_model.php` は、**クーポン作成時の理想値（推奨初期値）を算出するモデル**です。

このファイルの役割は、たとえば以下です。

- クーポン作成時の推奨値を出す
- 初期割引率 / 最低割引率の候補を考える
- 粗利最大化に近い推移を試算する
- 利用率や利益面の観点から設計支援を行う

つまりこれは、**実運用中の現在割引率を決めるファイルではなく、設計時の推奨値・理想値を考えるためのモデル**です。

#### このファイルでやらないこと

- 今日の割引率を直接決める
- 残日数を計算する
- 次回いつ下がるかを計算する
- 使用時点の割引率を確定する

---

### 実行ロジック（`lib/coupon_schedule.php`）

`coupon_schedule.php` は、**保存済みプランに対して、いま現在の割引率を計算する実行ロジック**です。

このファイルが担当するのは、たとえば以下です。

- 開始日時と終了日時から総日数を出す
- 現在日時から経過日数を出す
- 現在割引率を計算する
- 残日数を出す
- 次回いつ割引率が下がるかを出す
- 使用時点の割引率を確定する

つまり、**実際のAPIや利用処理で使う「今の値」を1か所で決める中核**です。

---

### 業務判断ロジック（`lib/coupon_logic.php`）

`coupon_logic.php` は、**数値計算そのものではなく、業務上の状態判断**を扱います。

想定する役割：

- 発行可能かどうか
- 利用可能かどうか
- 使用済みかどうか
- 有効期限外かどうか
- どの状態メッセージを返すか

つまり、

- `coupon_schedule.php` = 数値計算
- `coupon_logic.php` = 業務判断

という分担です。

---

### 旧割引橋渡し層（`lib/coupon_discount.php`）

`coupon_discount.php` は旧構造の名残を含むファイルです。

現時点では、**既存コードを安全に移行するための橋渡し層として最小限残す方針**です。

今後の方針：

- フェーズ1では窓際化し、責務を薄くする
- 既存呼び出しを `coupon_schedule.php` に寄せる
- 参照箇所が整理できた段階で削除可否を判断する

最終的には、**不要であれば削除対象**です。

## データ構造の基本方針

### 1. `coupon_plans`
クーポンの設計ルールを持つテーブルです。

例：

- タイトル
- 商品名
- 説明
- 開始日時
- 終了日時
- 初期割引率
- 最低割引率
- 利用条件
- メモ
- 有効/無効

ここに保存されるのは、**クーポンの実体ではなく、早得ルールそのもの**です。

---

### 2. `coupons`
発行されたクーポン実体を持つテーブルです。

例：

- クーポンコード
- 対応するプランID
- 発行日時
- 使用日時
- 使用時割引率

ここで重要なのは、**使用時点で確定した割引率を保存する**ことです。

---

### 3. `usage_logs`
利用処理の記録を持つテーブルです。

例：

- 利用日時
- 利用クーポンID
- 利用時割引率
- 店舗ID
- スタッフID
- ユーザー識別子
- 補足メモ

このログは、後の分析や検証の基礎になります。

---

### 4. `view_logs`（将来または段階導入）
閲覧記録用のテーブルです。

PoC段階では必須ではありませんが、将来的には

- 閲覧数
- 利用率
- 割引率別利用傾向

を見るために追加候補です。

---

## APIの責務

このPoCでは、APIの責務を明確にします。

### `api/coupon_current.php`
現在公開中のクーポンプランに対して、

- 現在割引率
- 残日数
- 次回変化情報
- 表示用基本情報

を返すAPIです。

### `api/coupon_issue.php`
クーポンを発行するAPIです。

- クーポンコード生成
- クーポン実体作成
- 発行日時保存

を行います。

### `api/coupon_get.php`
発行済みクーポンの状態取得APIです。

- クーポンコード/IDに紐づく状態取得
- 現在の利用可否確認
- 必要に応じた表示情報返却

を行います。

### `api/coupon_use.php`
クーポン使用を確定するAPIです。

ここで最も重要なのは、

- 使用可能か確認する
- `coupon_schedule.php` で使用時点の割引率を算出する
- その値を確定する
- `used_discount_rate` に保存する
- 使用済み状態に更新する
- 利用ログを書き込む

ことです。

## 管理画面の考え方

管理画面は、**固定割引クーポンを作る画面ではなく、早得ルールを設定する画面**として扱います。

主な目的：

- プラン作成
- 推奨値モデルを参考にした初期値入力
- 公開期間設定
- 初期割引率 / 最低割引率設定
- 商品名 / 利用条件 / 説明の設定
- 有効/無効切替

管理画面で扱うのは、**使用済み値そのものではなく、時間変動ルール**です。

## 現時点の重要方針まとめ

- クーポン取得時固定は採用しない
- 未使用中も割引率は変動する
- 割引率は使用時点で確定する
- `coupon_model.php` は推奨値モデル
- `coupon_schedule.php` は実行ロジックの中心
- `coupon_discount.php` は段階的に窓際化し、不要なら削除する
- フロントは後回しにし、先にロジックとAPIを固める

## 今後の整理方針

コード構成は、最終的に以下のような美しい分担を目指します。

- `coupon_model.php`  
  = 推奨値モデル（理想値・推奨初期値の算出）

- `coupon_schedule.php`  
  = 現在割引率・残日数・次回変化の計算

- `coupon_logic.php`  
  = 発行可否・利用可否・状態判定

- `coupon_discount.php`  
  = 段階的に縮小し、不要なら削除

- `db.php`  
  = DB接続 + `coupon_plans` 基盤関数群

この分担により、**意味がぶれず、説明しやすく、段階的に整理しやすい構造**にしていきます。
